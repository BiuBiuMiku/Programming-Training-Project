# 铁路票务系统 —— 可行性分析与概要设计报告


# 一、可行性分析

## 1.1 项目背景
传统购票流程繁琐、余票信息不透明，本系统旨在解决上述问题，提供车次查询、智能中转推荐、在线购票、退改签等核心功能。项目采用 C/S 架构，后端使用 C++（Drogon 框架）实现高性能业务逻辑与图算法，前端使用 C#（WPF 框架）构建桌面客户端，双方通过 HTTP + JSON 通信。系统设计参照铁路客运规范（实名制、阶梯退票、改签限制等），并采用模块化架构，为后续扩展预留接口。

---

## 1.2 可行性结论（六大维度分析）

### 1.2.1 时间可行性
本次实训共 10 天，全天集中开发，无其他课程穿插。我们按阶段拆解了开发计划：
- 第 1~2 天：搭建开发环境、跑通 Drogon 和 WPF 的基础示例、确定前后端接口文档；
- 第 3~5 天：完成基础 CRUD（车次增删改查、简单购票退票）——这是必须保底的里程碑；
- 第 6~7 天：攻坚 Dijkstra 中转算法和并发锁票（若进度滞后，第 8 天可补进）；
- 最后 2 天：前后端联调、编写测试用例、修复缺陷、准备答辩材料。

**主要风险**：Dijkstra 算法和并发锁票的实现难度较高，可能挤占后期联调时间。  
**应对策略**：第 5 天设置中期进度检查点，若算法进展不顺，将"智能中转"降级为选做项，优先保证直达购票和退票功能的稳定可用。

### 1.2.2 技术可行性
团队成员已修读 C 语言课程，对指针、结构体、文件操作等概念不陌生。C++ 和 C# 虽是新语言，但语法基础与 C 一脉相承，Drogon 和 WPF 框架均有完善的官方文档和社区示例可供参考。

**核心技术点评估**：
- 后端：邻接表存储图结构、Dijkstra 最短路径计算、JSON 序列化/反序列化，均为库函数调用层面的工作，不涉及底层操作系统编程；
- 前端：HttpClient 异步请求 + WPF 数据绑定，参照官方 MVVM 模板即可实现；
- 数据层：采用内存数据结构 + JSON 文件持久化，避免引入数据库带来的额外学习成本。

**风险点**：Drogon 的编译配置对新手上手有一定门槛，WPF 的跨线程 UI 更新容易踩坑。  
**兜底方案**：遇到框架级问题及时向教师/助教求助，不自行死磕；若编译长期无法解决，可降级为控制台演示程序，核心算法逻辑不受影响。

### 1.2.3 项目难度可行性
项目功能按"必做"和"选做"两个层次划分，兼顾不同能力水平的同学：

| 层次 | 功能范围 | 技术要求 | 达标标准 |
|------|----------|----------|----------|
| **必做** | 登录/注册、车次 CRUD、按车次/站点查询、下单扣余票、退票退款 | 文件读写 + 数组/链表操作，约 1500~2000 行/人 | 人人可独立完成 |
| **选做** | 多条件模糊查询、Dijkstra 智能中转推荐、阶梯退票手续费、改签事务 | 图算法 + 并发锁，需额外学习 | 学有余力者选做，加分项 |

**策略**：优先集中力量完成必做功能，确保系统"可用"；若前期进度理想，再向选做功能分配人力。

### 1.2.4 人力可行性
采用 4 人小组开发模式，分工如下：
- **后端开发（2人）**：负责 Drogon 服务搭建、业务逻辑与算法实现、数据文件管理，核心代码须相互 Review；
- **前端开发（1人）**：负责 WPF 界面设计、数据绑定、接口调用，按后端 API 文档进行对接；
- **测试与文档（1人）**：负责测试用例设计、集成测试执行、操作手册与答辩 PPT 编写。

**协作机制**：每日站会同步进度；后端接口未就绪时，前端可使用 Mock 数据推进界面开发，避免相互阻塞。教师与助教全程提供技术支持。

**潜在风险**：若某位同学因故无法参与，由同组其他成员按既定分工接管其任务，确保进度不受影响。

### 1.2.5 硬件与环境可行性
实训地点为校内标准化教室，电力、网络等基础设施完备。全体成员均自备笔记本电脑，可安装 Visual Studio 2022、Git 等开发工具，操作系统统一为 Windows，无跨平台兼容性问题。

### 1.2.6 法律与合规可行性
开发工具均为社区版或开源授权（Visual Studio Community、Drogon、.NET 8），无版权风险。系统使用完全虚构的车次数据与模拟用户信息，不涉及真实个人隐私或商业数据，纯教学用途，不进行任何商业部署。

---

### 1.2.7 综合结论
基础版功能（CRUD + 直达购退票）有充分把握完成，其技术难度均在课内知识覆盖范围内。进阶版功能（智能中转 + 并发锁票）存在一定挑战，主要取决于算法调试进度，预估有 70% 的成功率。我们已预留 2 天缓冲期，并约定：若第 7 天算法仍未跑通，则放弃中转功能，将剩余精力集中于系统联调和缺陷修复，确保交付一个稳定可用的基础版本。项目整体具备可行性。


# 二、系统总体架构（技术全景）

## 2.1 物理架构
- **后端（C++）**：基于 Drogon 框架提供 RESTful API，内置最短路计算引擎。站点拓扑图在内存中以邻接表形式维护，车次、订单、用户等数据以 JSON 文件作为持久化存储，服务启动时加载，运行时变更按策略回写文件。
- **前端（C#）**：基于 .NET 8 的 WPF 桌面应用，采用 MVVM 设计模式实现界面与业务逻辑的解耦，通过 HttpClient 以异步方式调用后端 REST 接口。
- **数据层**：当前采用内存数据结构（unordered_map、vector）+ JSON 文件持久化方案，满足设计阶段的演示与测试需求。数据访问接口已做抽象，后续可替换为 SQLite 或 MySQL 等关系型数据库而不影响上层业务逻辑。


## 2.2 逻辑分层架构

| 层级 | 后端（C++）实现 | 前端（C#）实现 | 职责 |
|------|----------------|---------------|------|
| **表现层** | 无（仅提供 API 端点） | XAML 窗口/控件 + ViewModel | 用户交互、数据绑定、前端输入格式校验 |
| **控制层** | Drogon 路由控制器，负责 URL 解析、参数提取与权限注解校验 | 前端页面导航器 + Command 绑定 | 请求路由、参数合法性初筛、访问权限验证 |
| **业务层** | C++ Service 单例类，封装购票、退票、改签、路径规划等核心逻辑 | C# ServiceProxy 类，封装 HttpClient 调用，按后端接口一一映射 | 核心算法执行、事务一致性保证、业务规则校验 |
| **数据层** | C++ 内存数据结构 + 文件 I/O 工具类 | 无（前端不维护业务状态） | 图结构存储、对象序列化/反序列化、进程重启后的数据加载与恢复 |


# 三、功能模块详细划分（四大核心模块）

| 模块编号 | 模块名称 | 核心职责 | 关键技术点（C++/C#） |
|----------|----------|----------|----------------------|
| **M1** | 用户登录与权限管理 | 注册、登录、JWT 身份认证、基于角色的访问控制（普通用户/管理员）、会话生命周期管理 | C++：JWT 生成与过期校验，密码采用 bcrypt 或 SHA-256 存储哈希值；C#：密码框使用 SecureString 保障内存安全，Token 加密后存储于本地配置文件 |
| **M2** | 车次与线路管理 | 车次信息的增删改查、时刻表维护、站点拓扑图的构建与动态更新 | C++：启动阶段解析 CSV 构建邻接表，管理员操作后同步更新内存图结构；C#：DataGrid 实现车次列表展示与编辑，支持通过文件导入向导批量录入车次数据 |
| **M3** | 票务核心业务（含智能中转） | 按出发站/到达站/日期查询余票、Dijkstra 最短路算法生成中转推荐方案、下单时的原子锁票与出票 | C++：Dijkstra/分层枚举实现路径搜索，查询时生成余票快照缓存；下单操作采用内存互斥量保护余票变量，保证并发场景下的数据一致性；C#：异步调用查询接口，搜索结果以车次方案卡片列表形式展示 |
| **M4** | 退票与改签 | 阶梯退票手续费计算、改签时旧票释放与新票占用的原子性换票操作 | C++：退票按距离开车时间查表计算手续费率；改签将"退旧票+买新票"组合为原子事务，使用互斥锁保证两步操作的不可分割性；C#：操作前弹出二次确认对话框，提交后展示进度提示并禁用重复点击 |
