# 一、可行性分析

## 1.1 项目背景
本系统旨在解决传统购票中的效率与体验问题，提供车次查询、智能中转推荐、在线购票、退改签等核心功能。项目采用 C/S 架构，后端使用 C++（Drogon 框架）实现高性能业务逻辑与图算法，前端使用 C#（WPF 框架）构建桌面客户端，双方通过 HTTP + JSON 通信。系统设计参照铁路客运规范（实名制、阶梯退票、改签限制等），并采用模块化架构，为后续扩展预留接口。

---

## 1.2 可行性结论（六大维度分析）

### 1.2.1 时间可行性
- **条件现状**：本次实训为 **10 天集中训练**，学校统一安排全天实训时间，开发周期连续、无碎片化中断。项目按阶段拆解为：基础环境搭建（1天）、数据结构与接口设计（2天）、核心业务编码（3天）、算法优化与调试（2天）、测试与文档（1天）、答辩准备（1天），每日有明确交付目标。
- **适配分析**：集中化时间安排可保证开发状态连贯；分阶段任务与每日增量提交机制，能及时发现进度偏差；设置 **中期检查节点（第5天）** 用于纠偏，并预留 **2天缓冲期** 用于改错和优化。即使部分同学前端或后端进度稍慢，也可通过小组内互助追赶。
- **结论判定**：本次实训时间规划科学紧凑，完全覆盖项目全流程，时间维度 **完全可行**。

### 1.2.2 技术可行性
- **条件现状**：项目技术栈为 **C++（Drogon） + C#（WPF）**，两者均基于大一已接触的 C 语言基础（指针、结构体、文件操作等）。Drogon 和 WPF 均为成熟框架，具备完善的官方文档与社区支持。
- **适配分析**：虽然框架本身有一定学习曲线，但学生可基于模板快速上手。核心技术点（邻接表、Dijkstra 算法、JSON 解析、文件 I/O）均为 C/C++ 基础知识的延伸，无超纲内容。此外，技术难度分层设计 —— 基础功能（车次增删改查、简单查询）人人可完成，进阶功能（智能中转算法、事务性退改签）由教师指导分组实现。
- **结论判定**：技术栈贴合学生已有基础，配套教学与模板支持到位，技术维度 **完全可行**。

### 1.2.3 项目难度可行性
- **条件现状**：项目功能分为 **基础版（必做）** 和 **进阶版（选做）**。基础版包含用户登录、车次管理（增删改查）、按车次查询、在线购票（余票扣减）、退票（简单退款）；进阶版包含多条件模糊查询、中转站智能推荐（Dijkstra）、阶梯退票手续费计算、改签事务操作。
- **适配分析**：基础版聚焦增删改查与文件持久化，逻辑清晰，代码量适中（约 1500~2000 行/人），零基础学生可通过复制模板、填充函数完成；进阶版增加算法与事务逻辑，面向学有余力的学生，不强制全员完成，不影响整体进度。项目剔除了网络编程（HTTP 通信由框架封装）、数据库等超纲难点，所有算法实现均基于教材中的图与排序内容。
- **结论判定**：难度梯度合理，兼顾“保底达标”与“拔高提升”，不会出现全员无法完成或难度过低的情况，难度维度 **具备可行性**。

### 1.2.4 人力可行性
- **条件现状**：实训采用 **4 人小组** 开发模式，每组分工为：后端开发（2人，负责 C++ 业务逻辑与算法）、前端开发（1人，负责 WPF 界面与交互）、测试与文档（1人，负责用例设计、集成测试、项目文档）。
- **适配分析**：分工明确，各角色职责清晰，可避免工作量堆积；小组内部实行每日站会同步进展，互相 Code Review，杜绝“搭便车”现象。教师与助教覆盖开发、调试、文档、答辩全流程，能及时介入解决小组阻塞问题。人力配置满足项目开发需求。
- **结论判定**：团队协作机制健全，师资保障充分，人力维度 **完全满足**。

### 1.2.5 硬件与环境可行性
- **条件现状**：实训地点为校内标准化教室，电力、空调、投影、白板等设施齐全；全体学生自备笔记本电脑，均可安装 **Visual Studio 2022（支持 C++/C#）**、**Git**、**Visual Studio Code** 等工具，设备性能满足开发要求；校园网络稳定，可支持代码仓库同步与在线文档查阅。
- **适配分析**：软硬件环境支持编码、编译、调试、测试、答辩展示全流程，无兼容性或性能瓶颈。
- **结论判定**：实训场地与设备条件完备稳定，环境维度 **完全可行**。

### 1.2.6 法律与合规可行性
- **条件现状**：所用开发工具（Visual Studio Community、Git、Drogon、.NET 8）均为开源或免费教学授权，无版权风险；项目使用 **虚构车次数据** 和 **模拟用户信息**，不涉及真实个人隐私或商业数据。

- **结论判定**：项目全程合法合规，风险可控，合规维度 **完全可行**。

# 二、系统总体架构（技术全景）

## 2.1 物理架构
- **后端（C++）**：基于 Drogon 框架提供 RESTful API，内置最短路计算引擎，维护内存中的站点拓扑图，数据以 JSON 文件持久化。
- **前端（C#）**：.NET 8 WPF 应用，采用 MVVM 模式，通过 HttpClient 调用后端接口，界面与业务逻辑分离。
- **数据层**：设计阶段使用内存数据结构 + JSON 文件模拟数据库，后期可无缝替换为 SQLite 或 MySQL。

## 2.2 逻辑分层架构

| 层级 | 后端（C++）实现 | 前端（C#）实现 | 职责 |
|------|----------------|---------------|------|
| **表现层** | 无（仅 API 控制器） | XAML 窗口/控件，ViewModel | 用户交互、数据绑定、输入校验 |
| **控制层** | Drogon 路由控制器 | 前端导航/命令绑定 | 请求路由、参数解析、权限验证 |
| **业务层** | C++ Service 类（单例） | C# ServiceProxy（调用 API） | 核心算法、事务逻辑、数据校验 |
| **数据层** | C++ 内存数据结构 + 文件 I/O | 无（仅接收后端数据） | 图存储、对象持久化、缓存 |
