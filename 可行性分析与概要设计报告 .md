# 一、可行性分析
## 1.1 项目背景
本系统旨在解决传统购票中的低效与信息不透明问题，提供车次查询、智能中转推荐、在线购票、退改签等核心功能。项目采用 C/S 架构，后端使用 C++ 实现高性能业务逻辑与图算法，前端使用 C#（WPF）构建桌面客户端，双方通过 HTTP + JSON 通信。

## 1.2 可行性结论
- 技术可行 ：C++ 具有极高的计算效率，适合最短路算法；C# WPF 提供丰富的桌面 UI 组件；
- 业务可行:所有流程参照铁路客运规范（实名制、阶梯退票、改签限制等），规则明确。
- 扩展可行：模块化设计便于后续升级为分布式集群或增加数据持久化层。

# 二、系统总体架构（技术全景）
## 2.1 物理架构
- 后端（C++）：基于 Drogon（或 oat++）框架提供 REST API，内置最短路计算引擎，维护内存中的站点拓扑图。
- 前端（C#）：.NET 8 WPF 应用，使用 MVVM 模式，通过 HttpClient 调用后端接口。
- 数据层：设计阶段仅定义抽象数据接口（可映射为 JSON 文件或 SQLite），无需真实数据库。

## 2.2 逻辑分层架构

| 层级 | 后端（C++）实现 | 前端（C#）实现 | 职责 |
|------|----------------|---------------|------|
| **表现层** | 无（仅 API 控制器） | XAML 窗口/控件，ViewModel | 用户交互、数据绑定、输入校验 |
| **控制层** | Drogon 路由控制器 | 前端导航/命令绑定 | 请求路由、参数解析、权限验证 |
| **业务层** | C++ Service 类（单例） | C# ServiceProxy（调用 API） | 核心算法、事务逻辑、数据校验 |
| **数据层** | C++ 内存数据结构 + 文件 I/O | 无（仅接收后端数据） | 图存储、对象持久化、缓存 |

## 三、功能模块详细划分（四大核心模块）

| 模块编号 | 模块名称 | 核心职责 | 关键技术点（C++/C#） |
|----------|----------|----------|----------------------|
| **M1** | 用户登录与权限管理 | 身份认证、RBAC 权限控制、会话管理 | C++：JWT 生成与校验；C#：SecureString 处理密码，Token 本地加密存储 |
| **M2** | 车次与线路管理 | 维护车次、时刻表、票价，构建铁路拓扑 | C++：邻接表构建，Excel/CSV 解析库；C#：DataGrid 绑定，文件导入向导 |
| **M3** | 票务核心业务（含智能中转） | 余票查询、最短路中转推荐、原子锁票、出票 | C++：Dijkstra/分层枚举算法，内存图快照；C#：异步搜索界面，方案卡片展示 |
| **M4** | 退票与改签 | 退改签规则计算、原子性换票 | C++：事务性操作（内存锁）；C#：确认对话框，进度提示 |
